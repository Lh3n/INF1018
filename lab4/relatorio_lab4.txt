1) Nesses programa, ao atribuir char c = 150, como char é com sinal em muitas
implementações, o valor armazenado internamente é 150 - 256 = -106, mas a
representação em 8 bits é 10010110 (0x96). Para short s = -3, em 16 bits complemento de
dois, o valor fica 11111111 11111101 (0xFFFD). Já int i = -151, em 32 bits complemento de
dois, resulta em 11111111 11111111 11111111 01101001 (0xFFFFFF69). Assim, o dump
deve mostrar, em ordem de memória, os bytes correspondentes a essas representações,
confirmando os cálculos.

2) Nesse programa, o short l = -32765 em complemento de dois de 16 bits gera 10000000
00000011 (0x8003), enquanto unsigned short k = 32771 ocupa os mesmos 16 bits
10000000 00000011, mas interpretados como valor positivo sem sinal. Por isso, na
impressão, l aparece como -32765 e k como 32771, embora ambos tenham a mesma
sequência binária interna. O dump evidencia isso ao exibir bytes idênticos para l e k,
mostrando como apenas o tipo muda a interpretação do mesmo padrão de bits.

3) O programa lê dois argumentos da linha de comando: um número em hexadecimal e
a posição de um byte; primeiro, converte a string hexadecimal em inteiro com
`string2num`, validando cada caractere, e depois usa a função `xbyte` para extrair
o byte desejado por meio de deslocamento de bits, aplicando extensão de sinal ao
convertê-lo para inteiro; por fim, imprime o valor resultante em formato hexadecimal
e decimal, encerrando com erro caso os argumentos estejam incorretos ou o número seja
inválido.

4.1) Como x é um signed int, 0xffffffff equivale a -1,
que é menor que 2.

4.2) Como x é um unsigned int, 0xffffffff equivale a 4294967295,
que é maior que 2.

4.3) Como y é um unsigned int e x é um signed int, x é convertido
para o tipo de maior grandeza, logo o valor de x unsigned int equivale
a 4294967295 que é maior que 2.

5)Como sc é um signed char que vale -1, que em 8 bits é representado em
complemento de dois como 11111111 (0xFF), ao atribuir esse valor a uma variável
unsigned int ui, ocorre primeiro a promoção com extensão de sinal para 32 bits, resultando
em 0xFFFFFF. Em seguida, esse mesmo padrão de bits é
reinterpretado como sem sinal, o que equivale a 4294967295 (0xFFFFFFFF). Assim, embora
sc represente o valor -1, após a atribuição, ui armazena o valor máximo possível para um
unsigned int. O programa de verificação em C confirma esse comportamento ao imprimir
ui como 4294967295 em decimal e 0xFFFFFFFF em hexadecimal.
